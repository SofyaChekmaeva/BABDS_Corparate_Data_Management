## Проект: SupportRAG - умная система поиска и генерации ответов на основе обращений в службу поддержки

## Проблема
В крупных компаниях операторы службы поддержки ежедневно обрабатывают огромный массив повторяющихся обращений клиентов. В результате возникает ряд проблем, из которых ключевыми можно выделить следующие:
- Высокая нагрузка на сотрудников;
- Увеличение времени ответа;
- Обработка каждого нового обращения независимо от истории (могут быть похожие по теме обращения).

Если сформулировать проблему более емко: **нет быстрого доступа к релевантным решениям из прошлых обращений клиентов**, что замедляет работу и снижает качество сервиса

## Цель проекта
Разработать прототип RAG-системы, которая по тексту нового обращения клиента:
1. **Находит** похожие решённые тикеты из базы знаний;
2. **Генерирует** проект ответа оператору на основе найденных решений.

Цель с точки зрения выполнения учебного задания - продемонстрировать полный пайплайн RAG:
- Загрузка и индексация текстовых данных;
- Поиск релевантных документов;
- Генерация ответа с использованием контекста.

## Ключевые стейкхолдеры проекта:
- **Операторы поддержки** — заинтересованы в быстром поиске готовых решений по похожим проблемам;
- **Руководитель службы поддержки** — заинтересован в аналитике по типовым проблемам и улучшении базы знаний;
- **Аналитики по клиентскому опыту** — заинтересованы в выявлении частых сценариев обращений;

## Данные (источник и масштаб)
**Источник:** [Customer Support Tickets Dataset (~200K записей)](https://www.kaggle.com/datasets/mirzayasirabdullah07/customer-support-tickets-dataset-200k-records)
- **Формат:** CSV, 73.8 MB
- **Количество записей в исходном датасете:** 200 000
- **Ключевые поля:**
  - `issue_description` — текст проблемы клиента (основной источник для поиска и извлечения данных)
  - `resolution_notes` — ответ поддержки (будет использоваться при генерации)
  - `Ticket Category` — категория проблемы (Login Issue, Payment Problem и др.)
  - `Ticket ID` — уникальный идентификатор

После чанкинга (нарезки текста) ожидается **10 000 – 15 000 чанков**

## Критерии успеха
Система считается успешно реализованной, если в ней настроены:

1. **Индексация:** данные загружены, разбиты на чанки и проиндексированы (TF‑IDF + косинусное сходство).
2. **Поиск (retrieval):**
   - Для тестового запроса «Не могу войти в аккаунт, пишет неверный пароль» система находит хотя бы один чанк с категорией `Login Issue` или `Account Suspension`.
   - `recall@5` на размеченном подмножестве > 0.6.
3. **Генерация (generation):**
   - Система выдаёт связный ответ, который ссылается на найденные чанки;
   - На запросы не по теме (например, "Как приготовить борщ? :)") система отказывается отвечать;
4. **Оценка:** реализованы метрики `recall@k`, `MRR`, `precision@k`;
5. **Ноутбук:** отчёт `homework/report.ipynb` содержит визуализации распределения категорий, примеры поиска и вывода.
